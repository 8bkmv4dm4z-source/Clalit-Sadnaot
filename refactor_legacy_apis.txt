Understand the current implementation
Server-side

Delete route – The deleteUser controller accepts a :id parameter, interprets it as an entityKey, resolves it to a user via resolveEntity, then deletes the user using the resolved _id. It also unregisters the user and all family members from workshops and logs the action. The function passes the entityKey or hashedId into the audit log
.

Routes – Both /api/users/:id and /api/profile/:id point to deleteUser, so deletion currently hinges on the entityKey.

Utilities – Helper functions (e.g., resolveEntity, resolveEntityByKey) convert an entityKey or hashed id to a Mongo _id.

Logging – Audit logs record the subjectKey, adminId and entityId using entityKey or hashed id
.

Client-side

Profile context – The deleteEntity function expects an entityKey. It calls DELETE /api/profile/{entityKey}, refreshes the profile list and returns success/failure
.

UI components – The handleDeleteEntity helper obtains the entityKey from getEntityIdentifiers(row) and calls deleteEntity({ entityKey })
. The UI uses entityKey to build row keys and deduplicate entities.

2. Objectives of the refactor

Unify identity – Use the Mongo _id (ObjectId) as the single identifier for users and family members across all API endpoints and client calls.

Remove legacy entityKey from route parameters – entityKey may still exist on documents for logging or compatibility, but it should not appear in API paths or client calls.

Fix the bug in the DELETE user route – Ensure the route properly identifies the user by _id, unregisters all related workshop registrations, deletes the document, and logs the operation.

3. Server-side refactor plan

Introduce _id‑based routing

Add new route definitions such as router.delete("/users/:userId", …) and router.delete("/profile/:userId", …) where userId must be a valid ObjectId. Deprecate the existing :id routes that accept entityKey.

Update the API documentation to reflect that all IDs must be Mongo _id values.

Remove resolveEntity for deletion

In the deleteUser controller, stop calling resolveEntity(entityKey…). Instead, treat req.params.userId as the Mongo _id. Validate it with mongoose.Types.ObjectId.isValid.

Fetch the user directly:

const user = await User.findById(userId).select("userWorkshopMap familyWorkshopMap");
if (!user) return res.status(404).json({ success: false, message: "User not found" });


Use user._id in all subsequent operations (unregistering workshops, deleting the user).

Update unregister helpers

The existing unregisterUserFromWorkshop and unregisterFamilyFromWorkshop functions already accept _id. Ensure they continue to work unchanged.

Adjust audit logging

Change the audit log payload to reference _id rather than entityKey:

await safeAuditLog({
  eventType: AuditEventTypes.ADMIN_USER_DELETE,
  subjectType: "user",
  subjectId: user._id.toString(),
  actorId: req.user?._id?.toString(),
  metadata: { action: "user_delete", adminId: req.user?._id?.toString(), entityId: user._id.toString(), ip: req.ip },
});


Keep entityKey or hashedId in the metadata only if you need backward‑compatibility in the logs.

Modify other routes to accept _id

getUserById, getEntityById, updateEntity, getUserWorkshopsList, and search endpoints should accept _id and, for family entities, an optional familyId query parameter. Remove use of resolveEntityByKey or adapt it to resolve only _id.

For updates, require req.body.userId or req.body.familyId rather than an entityKey.

Deprecate entityKey

Leave the pre‑save hook that generates entityKey in the User model for now (it may still be useful for logs or external systems), but mark it deprecated.

Remove any unique indices that enforce entityKey uniqueness if they are no longer needed.

4. Client-side refactor plan

Update identity utilities

Refactor client/src/utils/entityTypes.js so that getEntityIdentifiers() returns { userId: <_id>, familyId, parentUserId, isFamily }. Remove entityKey from its output.

Modify withEntityFlags() to include _id and parentId for family members.

Modify ProfileContext

Change the deleteEntity function to accept { userId } instead of { entityKey }, and call:

const res = await apiFetch(`/api/profile/${encodeURIComponent(userId)}`, { method: "DELETE" });


Similarly, adjust updateEntity to send { userId, updates } or { userId, familyId, updates }.

Adapt UI components

In AllProfiles.jsx, replace entityKey usage with _id:

const handleDeleteEntity = async (row) => {
  const { userId } = getEntityIdentifiers(row);
  …
  const response = await deleteEntity({ userId });
};


Update buildEntityKey(row) to use _id as the primary key (e.g., return row._id || row.id).

Workshop context

Update functions like unregisterEntityFromWorkshop, getUserWorkshops, etc., to accept { userId, familyId } instead of entityKey/familyEntityKey.

Ensure modals and lists display the correct identifiers.

Migration support

For a transition period, the frontend can read both _id and entityKey from server responses but should always send _id to the API. The server can still resolve entityKey via resolveEntity as a fallback but should log a deprecation warning.

5. Addressing the deletion bug

The current bug likely stems from the reliance on resolveEntity and entityKey, which can mis‑resolve or fail when a user has no entityKey (e.g., new documents) or when entityKey is duplicated. Switching to _id removes this ambiguity.

The deletion routine should also ensure all asynchronous calls complete. To avoid partial deletions, wrap workshop unregistrations in Promise.all and handle errors:

const unregisterUserPromise = Promise.all((user.userWorkshopMap || []).map(wid =>
  unregisterUserFromWorkshop({ workshopId: wid, userId: user._id })
));
const unregisterFamilyPromise = Promise.all((user.familyWorkshopMap || []).flatMap(entry =>
  (entry.workshops || []).map(wid =>
    unregisterFamilyFromWorkshop({ workshopId: wid, parentUserId: user._id, familyId: entry.familyMemberId })
  )
));
await Promise.all([unregisterUserPromise, unregisterFamilyPromise]);


After unregistrations, call User.deleteOne({ _id: user._id }) and log the operation as shown above. Return a clear success message.

6. Testing and deployment

Write integration tests for all CRUD routes, ensuring that:

Valid _id values delete, update and fetch users/family members correctly.

Invalid or malformed IDs return 400 Bad Request.

All workshop registrations are removed when deleting a user.

Update client tests to ensure deleteEntity({ userId }) works and that UI components display and handle _id correctly.

Deploy the changes to a staging environment, gradually remove support for entityKey in older clients, and then deploy to production.
