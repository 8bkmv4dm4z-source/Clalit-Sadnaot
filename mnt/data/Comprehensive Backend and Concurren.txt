Comprehensive Backend and Concurrency Audit for my‑react‑app‑public
Executive Summary

This audit examined the Node/Express backend that powers the workshops platform. The review focused on API endpoints, database models and update patterns, concurrency handling, invariants enforcement, idempotency, and failure modes. The codebase follows many best practices: JWT‑based authentication, strict request validation, role‑based authorization, separation of admin/user/public scopes, and use of MongoDB with Mongoose for persistence. The major functional areas reviewed were workshop management, registration and waitlist handling, user management, and authentication. While the design is well structured, a number of concurrency bugs and invariant gaps were found, especially around workshop registration and waitlist management. These issues can cause over‑booking, duplicate registrations and inconsistent state under high load or network retries. Fixes are outlined below. Once addressed, the system can be made production‑ready and resilient to partial failures and concurrent users.

Critical Findings (Must Fix Before Production)

Waitlist capacity filter bug – The addEntityToWaitlist function uses a filter on a non‑existent field waitingList.waitingCount to enforce the maximum waiting list size. Because waitingCount is not in the schema, the filter always passes, allowing unlimited waitlist entries. This race condition is severe; simultaneous waitlist registrations could oversubscribe the waiting list.

Non‑atomic unregistration and counts – unregisterEntityFromWorkshop modifies arrays and recalculates participantsCount outside of a transaction. Concurrent unregister and register calls can leave participantsCount incorrect and create gaps for oversubscription.

Parallel registration race – registerEntityToWorkshop correctly uses atomic conditional updates on participantsCount and checks for existing participants. However, workshopRegistration.js exports alternative registration functions that directly modify arrays using Workshop.findById and push/save (non‑atomic). If any part of the codebase still calls these legacy functions, they could bypass capacity checks entirely. These should be removed or updated.

Missing idempotency keys – Many POST/DELETE endpoints (e.g., register/unregister, waitlist add/remove, user creation) are not idempotent. Retrying a registration or unregistration can create duplicates or negative counts. There is no idempotency token or de‑duplication mechanism at the API layer.

Lack of database transactions – Multi‑entity updates (updating Workshop and corresponding User maps) are split into separate operations. Partial failures (e.g., crash after updating Workshop but before updating User) can leave data inconsistent. MongoDB transactions should be used or compensating operations must be built.

Recommendation

For endpoints that mutate state, support optional Idempotency-Key headers (similar to Stripe or AWS). Store a short‑lived token keyed by user and idempotency key that caches the result. On retry with the same key, return the previous response instead of performing the action again.

Failure Mode & Recovery Analysis

Database write succeeds but secondary update fails – In registerEntityToWorkshop, after successfully adding to Workshop, the code updates the User document. If this second update fails (e.g., network error), the workshop will reflect the registration but the user’s map will not. There is no rollback or retry. Similarly, unregisterEntityFromWorkshop updates both documents separately. Recommendation: Wrap cross‑collection mutations in a MongoDB transaction or implement a best‑effort compensating update triggered by a background job.

Database write succeeds but audit log fails – Audit logging occurs after database writes. If logging fails, the system still returns success. This is acceptable; however, logs could be lost silently. Consider capturing failures and sending them to an error queue for later reconciliation.

DB success but email/notification fails – The exportWorkshopExcel function sends an email after generating an Excel report. If the email service fails, the export might silently fail. The function catches errors and returns a 500 but does not attempt to rollback or flag the export as incomplete. Consider queuing email sending and marking the export status in DB.

Partial updates due to crashes – In any endpoint where multiple asynchronous operations occur (e.g., waitlist add, register), if the process crashes mid‑function, some operations may persist while others do not. Without atomic transactions or a reconciliation mechanism, inconsistent state persists. A watchdog job should periodically reconcile counts with actual array lengths and fix discrepancies.

Worker crashes during auto‑promotion – autoPromoteFromWaitlist loops until canAddParticipant() and shifts from the waitlist. If a crash occurs mid‑loop, some promotions may be applied, leaving the array partially processed. participantsCount is recalculated only on save. Recommendation: Use atomic updates to pop a single entry and insert into participants within the same transaction. Schedule auto‑promotion via a background worker that retries until no entries remain.

Performance & Scaling Risks

Unbounded lists – getAllWorkshops and getWorkshopParticipants implement pagination with limit and skip, which is good. However, the default limit values are high (100 participants). Ensure default reasonable limits and indexes on participantsCount, waitingList and startDate for sorting.

N+1 queries – Some endpoints fetch user documents and then map across them (e.g., getRegisteredWorkshops resolves workshopKey by fetching each workshop by id). While small sets are expected, consider using $lookup to join in one query for large user maps.

Search performance – searchWorkshops uses Atlas Search with fallback to regex. The fallback can be expensive for large collections; ensure indexes on searched fields and monitor query latency.

Write amplification – Each registration triggers an update on the workshop document, the user document, and emits audit logs. Under high concurrency, this can cause contention on the same document. Consider sharding or splitting waitlist into a separate collection.

Security & Authorization Risks

EntityKey vs. _id exposure – The controllers correctly avoid exposing Mongo _id and use hashed entityKey/workshopKey. This reduces enumeration attacks.

Authorization checks – assertOwnershipOrAdmin ensures that only owners or admins mutate data. However, some admin routes rely solely on middleware; ensure all controller functions recheck hasAuthority where appropriate.

IDOR protection – Because clients cannot provide object IDs directly, mass assignment vulnerabilities are mitigated. Input validation rejects forbidden fields.

Rate limiting – The server config applies global and write‑specific rate limiters, and uses per‑user or per‑IP keys. This helps protect against brute force and DoS.

Concrete Fix Proposals
1. Fix Waitlist Capacity Filter and Enforce Atomically

Replace the invalid waitingList.waitingCount filter with a proper check on the length of the waiting list. Because MongoDB cannot compare array length directly in a filter, store waitingListCount as a separate numeric field (similar to participantsCount) and update it atomically when pushing or popping from the array. For example:

// Add a numeric field waitingListCount to Workshop schema and keep it in sync via pre‑save hook.
// When adding to waitlist:
const filter = { _id: workshop._id, waitingListCount: { $lt: workshop.waitingListMax } };
const update = { $push: { waitingList: entry }, $inc: { waitingListCount: 1 } };
const updated = await Workshop.findOneAndUpdate({ ...filter, ...noDuplicateClause }, update, { new: true });


On removal:

await Workshop.updateOne({ _id: workshop._id }, { $pull: { waitingList: condition }, $inc: { waitingListCount: -1 } });


This ensures atomic enforcement of the waiting list maximum.

2. Use MongoDB Transactions for Multi‑Document Updates

Wrap registration and unregistration logic in a session:

const session = await mongoose.startSession();
try {
  session.startTransaction();
  const workshop = await Workshop.findOneAndUpdate(filter, update, { new: true, session });
  if (!workshop) throw new Error('No capacity or duplicate');
  // update user maps
  await User.updateOne(userFilter, userUpdate, { session });
  await session.commitTransaction();
} catch (err) {
  await session.abortTransaction();
  throw err;
} finally {
  session.endSession();
}


This makes state changes across collections atomic. All registration and waitlist functions should use transactions to avoid partial state.

3. Remove or Fix Legacy workshopRegistration Functions

Deprecate registerFamilyToWorkshop and registerUserToWorkshop from workshopRegistration.js or rewrite them to delegate to registerEntityToWorkshop with proper capacity checks. Audit the codebase to ensure they are not referenced by any route or background job.

4. Implement Idempotency Keys

Accept an Idempotency-Key header on write endpoints. Store processed keys in a short‑lived collection with the user id and hashed key. When a request arrives with a key that matches a stored one, return the previous response instead of executing the write again. This prevents duplicate registrations on network retries.

5. Introduce Reconciliation Job

Create a scheduled job that recalculates participantsCount and waitingListCount based on array lengths and fixes mismatches. This job can also detect orphaned references (e.g., a user in a workshop’s participants array who no longer has that workshop in their userWorkshopMap). The job should log anomalies and auto‑heal or raise alerts.

6. Improve Auto‑Promotion Logic

Refactor autoPromoteFromWaitlist to perform one promotion per call using an atomic update: pop the first waitlist entry and push it to participants or familyRegistrations within the same transaction. Then schedule the function via a background worker triggered whenever participantsCount is below maxParticipants and there are waitlist entries. Avoid loops in the request handler.

7. Additional Enhancements

Address Validation – The code already performs non‑blocking address validation with OpenStreetMap. Consider caching results to reduce external API calls.

Logging and Observability – Ensure structured logs include correlation IDs, user keys, and error details. Improve error messages when returning 400 vs. 500 to aid diagnostics.

Input Validation – validation.js sanitizes inputs; maintain strict Joi schemas and update as API evolves.

Final Production Verdict

The system demonstrates solid architecture, strict input validation, and good separation of concerns. However, several concurrency and idempotency issues threaten correctness and scalability. The most pressing problem is the broken waitlist capacity filter, which will cause oversubscription under concurrent usage. Additionally, the absence of transactions for multi‑document updates exposes the platform to partial failures. Implementing the fixes above—atomic counters for waitlist, MongoDB transactions, idempotency keys, reconciliation jobs, and refactoring legacy registration functions—will significantly improve robustness. Once addressed, the platform should handle concurrent users, network retries, partial outages, stale reads and malicious requests competently. Until then, the system should not be considered production‑ready.
